Advanced Concepts#

Graphs#

Graph is a represetation of how data and trigger dependency flows in Tabsdata.

Trigger Dependency Graph#

This graph represents how different tabfunctions are related to each other via triggers. A tabfunction can be triggered by any number of triggers from zero to n.

Data Dependency Graph#

This graph represents how different tabfunctions are related to each other via data dependencies. A tabfunction can have as input and be an input for any number of tabfunctions from zero to n.

Pristine State <<I like Consistent State. But consistency is a confused term.>> (Only available in Enterprise version.)#

As data is loaded into a Tabsdata system and is transformed by multiple tabfunctions, there will be cases where, as in the above Figure 1, a tabfunction (ie E) consumes data from two or more tabfunctions (ie C & D) which in turn consume data from upstream tabfunctions (ie A and B).

In this scenario, unless explicitly overruled, it is desirable that the consuming tabfunction (E) uses snapshots of the upstream tabfunctions (C & D) based on the latest upper stream tabfunctions (A and B). This is analogous to what in SQL is referred to as transactional consistency. We refer to this as a pristine state.

Every tabfunction within an instance has a notion of a pristine state which is a vector of table versions that together reflect a fully absorbed change from its upstream dependencies. Changes to tabfunctions are executed serially from a queue, with each change resulting in the publishing of a new pristine version that propagates to downstream tabfunctions.

Hence, a pristine state is a view of a tabfunction dependency graph where all the versions are derived from the same initial upstream snapshots.

If the inputs for a given tabfunction have 2 dependency graphs with an empty intersection, then the current versions of the inputs always are ‘pristine’ for the consuming tabfunction.

Execution Plan <<DOUBT: Is this needed?>>#

When a function is manually or programmatically triggered, an execution plan is created. This execution plan determines all the functions that need to be executed and the exact data version (existing, or to be created as part of this execution plan) each function involved in the execution plan needs.

Each execution plan is assigned a unique time, execution plan time, thus providing a strict order for execution plans.

Each execution plan will use its execution plan time to find the state of the system at that time and to create the resulting state after the execution completes. The resulting state exists as soon as the execution plan has been created, and can be used by subsequent execution plans to create their exit state. The functions will be executed only when all data required input data, its dependencies, exists.

An important aspect of this is that this enables execution of functions (from the same or different execution plans) in parallel as long as their input data is available.
